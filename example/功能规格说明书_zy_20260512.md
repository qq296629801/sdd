# 功能规格说明书

## 1. 文档概述

- **项目名称**：生产全过程正逆向追溯系统
- **文档版本**：v1.0
- **编写日期**：2026-05-07
- **文档目的与范围说明**：本文档旨在对“生产全过程正逆向追溯系统”的功能需求、数据模型、接口设计、UI交互、业务流程及非功能需求进行完整、详尽的描述和定义。本文档的服务对象为项目经理、系统架构师、前后端开发工程师、测试工程师及产品经理等所有项目干系人。文档范围覆盖从原料入库批次到成品出库的全链路正向追溯，以及从成品批次反向追查至原料消耗、过程检验、异常事件的全链路逆向追溯，包括支撑这些追溯能力所需的数据模型、API接口、用户界面和业务流程的详细设计。

## 2. 需求概述

### 2.1 项目背景
在现代离散制造与流程制造行业中，产品质量管控的核心挑战在于如何快速、准确地定位质量问题的根源并界定影响范围。当前，企业内部通常部署了ERP（企业资源计划系统）、MES（制造执行系统）、WMS（仓库管理系统）以及LIMS（实验室信息管理系统）或QMS（质量管理系统）等多个异构系统。这些系统在各自的专业领域内高效运转，但彼此间的数据往往是孤立的。原料在WMS中入库并生成批次号，在MES中被消耗并记录过程参数，在LIMS/QMS中完成检验并生成报告，最终在ERP中完成销售出库。当成品出现质量客诉时，质量部门需要花费数小时乃至数天，跨多个部门手动查阅不同系统的报表和单据，才能拼凑出该批产品的完整生产画像。同样，当原料检验发现异常时，要立即锁定所有使用了该批原料的在制品、半成品及已发出的成品，更是耗时且极易出现遗漏。这种低效的追溯模式不仅延长了质量问题的暴露时间，增加了召回成本，更可能因追溯不及时而对品牌信誉造成不可挽回的损害。因此，建立一套横跨原料、生产、检验、成品、销售全链路的统一追溯平台，实现从原料到客户以及从客户到原料的双向快速追溯，已成为企业保障产品质量、降低经营风险、满足合规审计的迫切需求。

### 2.2 项目目标
- **目标一：实现高效的正向追溯**：以原料批次号为核心入口，支持在3秒内查询出该批原料的检验报告、所有投入使用的生产工单、完整的加工过程数据（包含工序、设备、人员、工艺参数）、最终产出的所有产品批次，以及这些产品批次的销售客户信息。查询结果应展示完整的数据关联链路。
- **目标二：实现精准的逆向追溯**：以产品批次号为核心入口，支持在3秒内查询出该成品批次的最终检验报告、销售客户详情、关联的生产工单信息、工单生产过程中消耗的全部物料明细（精确到原料批次、用量、消耗时间）、完整的过程检验记录、关键工艺参数的实测值，以及生产过程中发生的所有异常事件记录。
- **目标三：建立统一的追溯数据模型**：设计并落地一套中心化的追溯数据模型。该模型应能整合来自WMS、MES、LIMS/QMS、ERP等至少4个异构系统的关键数据，将原料批次、工单、工序、设备、人员、质检记录、产品批次和客户信息进行唯一性关联，确保数据的完整性和一致性。

### 2.3 需求范围
- **功能范围清单（包含）**：
  1.  **正向追溯查询功能**：单一原料批次号精确查询，支持查询结果的多维度、可视化展示（树形或图形化拓扑图），展示内容包括原料检测报告、关联工单列表、加工过程详情、产出批次清单及对应销售客户信息。
  2.  **逆向追溯查询功能**：单一产品批次号精确查询，支持查询结果的多维度、可视化展示，展示内容包括成品检验报告、销售客户信息、关联工单详情、工单物料消耗明细、过程检验记录、过程工艺参数、以及异常事件记录。
  3.  **追溯数据整合与同步**：通过API接口或ETL任务，定期（准实时，延迟不超过5分钟）从WMS、MES、LIMS、ERP等源系统抽取、转换、加载并关联形成追溯所需的业务数据快照。
  4.  **权限管控**：基于角色的数据访问控制，实现不同部门人员（如质检、生产、销售、技术）查看被授权范围内的追溯信息。

- **功能范围清单（不包含）**：
  1.  从零开始构建新的WMS、MES、LIMS、ERP系统。
  2.  修改上游WMS、MES、LIMS、ERP系统的原始业务逻辑和数据库结构。
  3.  处理生产执行过程中的实时监控、报警或调度功能。
  4.  对供应商或客户信息的增删改查管理（只做数据同步和展示）。
  5.  移动端APP开发（本期仅涵盖Web端）。

### 2.4 术语定义
| 术语中文 | 术语英文（缩写） | 定义 |
| :--- | :--- | :--- |
| 原料 | Raw Material | 用于生产制造过程的原始物料，通常来自供应商采购。 |
| 原料批次号 | Raw Material Lot Number | 在原料接收和入库环节，由WMS系统生成或从供应商处接收的唯一性标识，用于区分不同时间、不同供应商到货的同类原料。编码规则为：“RM”+YYYYMMDD+6位流水号。 |
| 产品/成品 | Product/Finished Good | 生产制造过程完成后，经检验合格、包装完毕待销售的最终产物。 |
| 产品批次号 | Product Lot Number | 在成品包装或入库环节，由MES或WMS系统生成的唯一性标识。编码规则为：“FG”+生产工单号+4位序列号。 |
| 工单 | Work Order (WO) | MES中指导和记录生产任务的核心单据，定义了生产什么产品、数量多少、何时开始、使用何种物料和工艺路线。 |
| 加工过程 | Manufacturing Process | 从工单下达至成品产出期间，产品所经历的一系列工序活动，包括但不限于工序流转、设备操作、工艺参数执行、在制品状态变化。 |
| 正向追溯 | Forward Traceability | 从供应链上游向下游的追踪能力，在本项目中特指从原料批次开始，查找该原料被用在哪个工单，最终生产出哪些产品批次，并发货给了哪些客户。 |
| 逆向追溯 | Backward Traceability | 从供应链下游向上游的溯源能力，在本项目中特指从一个产品批次开始，反查其生产工单、过程中消耗的原料批次、源头供应商以及期间的所有质量和生产事件。 |
| 过程检验 | In-Process Inspection (IPQC) | 在产品生产过程中，对在制品或首件进行的质量检验，如首检、巡检、末检。 |

### 2.5 用户角色
| 角色名称 | 角色职责 | 权限概述 |
| :--- | :--- | :--- |
| 质量工程师 | 负责产品质量分析与客诉处理，是本系统的主要用户。 | 拥有正向追溯和逆向追溯的全部查询权限，可查看所有技术细节，包括检验报告、工艺参数、异常事件。 |
| 生产主管 | 负责监控生产状况，处理生产异常。 | 拥有正向和逆向查询权限，可查看工单信息、加工过程、过程检验、异常事件，但不能查看销售客户信息。 |
| 车间操作员 | 执行具体的生产任务。 | 仅拥有正向追溯的查询权限，且只能查看与本岗位相关的加工过程和物料信息，无权限查看检验报告和销售客户信息。 |
| 销售经理 | 负责客户关系维护与订单交付。 | 仅拥有逆向追溯的查询权限，可查询产品批次对应客户信息和成品检验报告，无法查看加工过程、工艺参数和成本相关数据。 |
| 系统管理员 | 负责系统的用户、权限、配置管理及数据监控。 | 拥有最高系统管理权限，包括角色权限分配、数据源配置、接口日志审计等，但不直接参与业务数据查询。 |

#### 3.1.3 UI 界面设计
- **页面布局**：正向追溯查询包含一个查询入口页和一个结果展示详情页。
  - **查询入口页**：顶部居中为页面标题“生产正向追溯”。标题下方是一个搜索栏，包含一个文本输入框和“查询”按钮。输入框占位符文字为：“请输入原料批次号，如 RM20260506123456”。页面其余部分为空状态。
  - **结果展示详情页**：查询成功后，跳转到此页面。页面采用自上而下的线性布局，清晰展示追溯链的每个环节。每个环节用卡片面板（Card）分隔。

- **交互流程说明**：
  1.  用户在搜索框中输入原料批次号。
  2.  点击“查询”按钮。按钮显示Loading状态，并禁用点击。
  3.  系统后台查询，数据返回后，按钮恢复正常。如果查询无结果，页面显示“未查找到与该批次号相关的记录”。
  4.  如果查询成功，页面渲染结果详情，从上到下依次展示：

      -   **③ 生产关联**：这是核心部分。按工单分组，展示每个消耗此原料的工单面板。每个工单面板内：
          -   头部：工单号、产品名称、消耗数量。
          -   下方有一个“加工过程”的时间轴组件。时间轴的每个节点代表一个工序，清晰展示工序顺序号、名称、时间区间、设备和操作员。点击节点，可展开显示该工序的详细工艺参数。

- **分页策略**：若产出批次或销售信息条数大于50条，采用前端懒加载（滚动到底部加载更多）的方式进行分页。

#### 3.1.4 业务流程设计
- **正常业务流程**：
  ```mermaid
  graph TD
      A[用户访问正向追溯页面] --> B[在搜索框输入原料批次号 RM.STL.001.20260506] ;
      B --> C{点击“查询”按钮} ;

      E --> F{校验通过？} ;

      J --> K[查询TRACE_PRODUCTION_RELATION表，获取关联的工单号] ;
      K --> L[循环查询 TRACE_MANUFACTURING_ORDER 获取工单详情] ;
      L --> M[查询 TRACE_PROCESS_RECORD 获取加工过程] ;
  ```

- **替代流程**：
  - `3a. 查询无结果`：如果在`I`处查询原料批次主表无记录，则直接返回HTTP 404错误，前端展示“未查找到与该批次号相关的记录”。
  - `3b. 数据部分缺失`：如果`J`处原料检验报告未找到，JSON响应中`inspectionReport`字段为`null`。前端在相应的卡片区域显示“暂无原料检验报告”文字提示。同理，加工过程或销售信息缺失时，均在该模块位置显示占位文本“暂无相关记录”，保持追溯链的完整性。
- **异常处理流程**：
  - `5a. 后端与上游数据库连接异常`：捕获数据库连接超时或查询超时异常，返回HTTP 500错误，前端显示“系统繁忙，请稍后再试”，并引导用户联系管理员。
  - `5b. 令牌过期或无效`：Spring Security拦截请求，返回HTTP 401状态码，前端捕获后跳转至登录页面。

#### 3.1.5 异常处理与边界条件
- **输入校验规则**：原料批次号`rawMaterialLot`字段，后端需进行严格校验，必须符合`^RM\d{14}$`的正则表达式，长度不够或字符不符的请求直接返回“参数格式错误”。
- **事务边界与回滚策略**：正向追溯查询为纯读操作，不涉及写操作，所有数据库操作在“读未提交（READ UNCOMMITTED）”或“读已提交（READ COMMITTED）”事务隔离级别下执行，不手动管理事务。性能优先。

- **权限控制**：仅限`ROLE_ADMIN`角色访问。

#### 3.3.3 UI 界面设计
为系统管理员提供一个“数据同步监控”页面，包含一个过滤栏和一个分页表格，显示最新的同步任务执行记录。列表展示任务名、状态（彩色标签）、开始时间、耗时、成功/失败记录数。点击失败的记录行，可弹出模态框查看错误日志详情。

#### 3.3.4 业务流程设计
- **数据同步流程**：
  ```mermaid
  graph TD
      A[定时任务或手动触发] --> B{选择任务}; 
      B --> C1[SYNC_RAW_MATERIAL_LOT];
      B --> C2[SYNC_INSPECTION_REPORT];
      B --> C3[SYNC_PRODUCTION_RELATION];
      C1 --> D1[连接源系统WMS，读取昨日至今新增/变更的数据];
      D1 --> E1{数据校验};
      E1 -- 失败 --> F1[记录失败日志到error_log字段，状态置为FAILED];
      E1 -- 成功 --> G1[批量写入本系统TRACE_RAW_MATERIAL_LOT表];
      G1 --> H1[更新日志: successRecords+N, status=SUCCESS];
  ```
- **处理策略**：所有同步任务均采用“先删后插”或“Upsert”策略。以批次号作为业务主键，确保数据同步的幂等性。

## 4. 非功能需求

### 4.1 性能要求
- **响应时间**：95%的正向/逆向追溯查询请求（数据量在百万级记录下），响应时间应小于3秒。最大响应时间不超过5秒。
- **吞吐量**：系统应能支撑每日至少100,000次追溯查询。
- **并发用户数**：支持至少50个用户同时执行查询操作，系统无报错或严重卡顿。

### 4.2 安全要求
- **认证授权**：采用基于JWT的无状态认证机制。用户登录成功后获得Token，所有API请求必须在Header中携带Token。严格遵循3.1.2和3.2.2中定义的基于角色的权限控制逻辑。
- **数据加密**：所有API通信强制使用HTTPS/TLS 1.3。数据库中的敏感字段（如客户名称、联系方式）在数据同步写入时进行AES-256加密存储，在前端展示时根据用户权限决定是否解密显示。
- **审计日志**：必须记录所有用户的查询操作。新建审计日志表`TRACE_AUDIT_LOG`，字段包含：操作人、IP地址、操作时间、查询类型（正向/逆向）、查询关键字（如批号）、查询结果状态（成功/失败）。查询操作不应记录返回的详细业务数据。
- **防注入**：后端全部采用预编译SQL（PreparedStatement）方式操作数据库，杜绝SQL注入风险。

### 4.3 可用性要求
- **系统可用率**：核心追溯查询服务应达到99.9%的可用率（每月停机时间不超过43分钟）。
- **容灾方案**：数据库必须配置为主从复制架构。应用服务器至少部署2个节点，并通过Nginx或云负载均衡器进行负载分发，实现单节点故障时的无感切换。
- **数据备份**：数据库每日凌晨3点进行全量物理备份，每6小时进行一次增量备份。备份文件保留30天，异地存储。

### 4.4 可扩展性
- **水平扩展策略**：应用服务器设计为完全无状态，可随时通过增加节点进行水平扩展。数据库层初期采用主从读写分离策略，读压力量大时，可从库数量可动态增加。
- **模块解耦要求**：正向追溯、逆向追溯、数据同步三大功能模块应作为独立的微服务或代码模块，API清晰定义，以便未来其中一个模块的技术栈升级或重构不影响其他模块。

### 4.5 兼容性
- **浏览器兼容**：必须支持最新版本的Google Chrome、Mozilla Firefox和Microsoft Edge。对于IE系列浏览器，不作兼容性要求。
- **接口版本管理**：API路径中包含版本号（如`/api/v1/`）。后续API如有破坏性变更，须发布新版本（如`/api/v2/`），同时保证旧版本在6个月内仍可继续使用，给消费者充足的迁移时间。
