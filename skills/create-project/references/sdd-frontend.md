# IIDP 前端与 UI 原型 SDD 规格模板

## 本文定位与使用时机

本文是”技术落地规格”模板，适用于**已知 IIDP 实现方式、需要输出节点树/数据源/绑定/事件/命令等技术细节**的场景。

| 场景 | 用哪个模板 |
|---|---|
| 从业务需求/用户故事生成前端规格（尚不确定实现方式） | 用 `skills/frontend/references/iidp-frontend-spec-doc/SKILL.md` 的模板 |
| 已有需求分析，需要输出节点树、selector、数据源等技术落地细节 | 用**本文**模板 |
| 需要页面级交互状态、响应式、可访问性验收 | 先读 `sdd-frontend-interaction.md`，再用本文补全技术细节 |

**两个模板的核心分工**：
- `iidp-frontend-spec-doc`：业务语言 → IIDP 实现建议（”要做什么、用哪种方式”）
- 本文（`sdd-frontend.md`）：IIDP 实现建议 → 技术落地规格（”节点怎么写、绑定怎么配”）

不要把两个模板混用或重复输出。确定实现方式后，用本文输出可直接指导代码生成的技术规格。

---

前端规格必须保留 UI 原型描述，但不能把原型描述直接等同于自定义前端开发。原型用于描述用户看到什么、怎么操作、每种状态怎么反馈；落地时仍然先判断标准模板/在线视图是否满足，再决定是否需要 hook、扩展视图或自定义 Vue2 组件。

如果用户明确要求”前端交互设计规格”、页面级交互状态、响应式行为或可访问性验收，优先读取 `sdd-frontend-interaction.md`，再回到本文补全 IIDP 前端规格。

生成 UI 原型描述前，必须先提示用户补充任意支持 MCP 的原型/设计/产品工具服务。该服务可以来自任何产品，只要能通过 MCP 暴露原型资源即可；不要限定或默认某一个原型产品。如果当前环境能列出 MCP 资源，优先使用 `list_mcp_resources` 或 `list_mcp_resource_templates` 查找用户指定的原型服务；用户提供资源 URI 后，用 `read_mcp_resource` 读取原型内容，并从中提取页面骨架、区域布局、用户流程、交互状态、弹窗/抽屉、空态和异常态。无法读取 MCP 服务时，允许使用截图、导出的原型说明或文字描述兜底，但必须在规格中标记“原型来源：待确认”。

````markdown
```text
[页面名称]
├── 顶部区域：[标题/面包屑/状态提示/全局按钮]

### 5.2 用户流程

```text
用户进入页面
  → 系统加载菜单和视图
  → 展示默认查询条件和列表数据
  → 用户执行 [查询/新增/编辑/删除/导入/导出/选择]
    → 成功：[刷新列表/关闭弹窗/展示提示/跳转页面]
    → 失败：[展示平台错误/字段校验/权限提示]
```

### 5.3 状态清单

| 状态 | 触发条件 | 展示/行为 | IIDP 承载方式 |
|---|---|---|---|
| 初始态 | 首次进入页面 | 默认查询条件、默认按钮状态 | 后端视图默认值/标准模板 |
| 加载态 | 请求 search/loadView/save | loading 或按钮禁用 | 标准模板/hook |
| 空态 | 查询结果为空 | 空数据提示，可保留新增入口 | 标准模板/扩展视图 |
| 校验失败 | 必填、格式、唯一性不满足 | 字段错误或业务提示 | `@Validate`/hook/form.validate |
| 权限受限 | 无菜单、按钮或服务权限 | 隐藏/置灰/后端拒绝 | auth/权限配置/服务校验 |
| 异常态 | 接口或服务异常 | 平台错误提示，不泄露敏感信息 | 后端异常/hook 错误加工 |

| 原型元素 | 首选 IIDP 承载方式 | 需要前端扩展的条件 |

| 数据表格 | `{model_name}_grid` | 复杂列渲染、特殊按钮、跨数据源聚合 |
| 新增/编辑表单 | `{model_name}_form` | 保存前特殊处理、动态显隐、复杂子表 |
| 树/上下表/子表 | 后端高级视图和标准模板 | 标准模板无法表达交互细节 |
````
