# 单元测试

以下文档为前端项目结合 AI 工具的单元测试指南，覆盖：环境搭建、项目结构与文件命名规范、常用运行命令、AI 辅助生成测试用例的实践、覆盖率统计与导出、未覆盖代码分析流程等。

## 一、环境搭建

### 1.1 创建并切换分支

在项目根目录执行以下命令，新建独立分支以隔离单元测试相关改动：

```bash
git checkout -b feat_unitTest
git branch   # 确认当前在 feat_unitTest
```

### 1.2 安装依赖

使用 `--legacy-peer-deps` 避免版本冲突：

```bash
npm install --save-dev jest@^29.5.0 jest-environment-jsdom@^30.2.0 jest-html-reporters@^3.1.7 jest-stare@^2.5.3 --legacy-peer-deps
```

### 1.3 package.json 新增脚本

在 `"scripts"` 字段中追加以下四条命令：

```json
"scripts": {
  "test:coverage" : "jest --config=jest.config.coverage.js",
  "test:report"   : "jest --config=jest.config.report.js",
  "test:unit"     : "jest --config=jest.config.coverage.js unitTest/apps/sie-iidp-xxx-web/views/xxx/xxx.unit.js",
  "test:func"     : "jest --config=jest.config.coverage.js -t \"$FUNC\" $FILE"
}
```

> **说明：**
> - `test:unit`：精确指定单个文件运行
> - `test:func`：使用 `-t` 参数按函数名过滤，两者均可组合使用

### 1.4 根目录创建文件夹

```bash
mkdir testReport   # 存放测试报告输出
mkdir unitTest     # 存放所有 .unit.js 测试文件
```

  // Mock localStorage
  if (!window.localStorage) {
    const localStorageMock = (function () {
      let store = {};
      return {
        getItem: jest.fn(key => store[key] || null),
        setItem: jest.fn((key, value) => { store[key] = value.toString(); }),
        removeItem: jest.fn(key => { delete store[key]; }),
        clear: jest.fn(() => { store = {}; }),
        key: jest.fn(i => Object.keys(store)[i] || null),
        get length() { return Object.keys(store).length; }

  // Mock sessionStorage

    observe: jest.fn(),
    unobserve: jest.fn(),

  }));

## 二、项目结构规范

### 目录对照关系

保持 `unitTest/` 目录结构与 `apps/` **完全镜像**，方便定位源码与测试文件的对应关系。

| 源码路径（apps/） | 对应测试路径（unitTest/） |
|---|---|
| `apps/sie-iidp-i18n-web/views/translationManagement/tm.js` | `unitTest/apps/sie-iidp-i18n-web/views/translationManagement/tm.unit.js` |
| `apps/sie-iidp-xxx/utils/helper.js` | `unitTest/apps/sie-iidp-xxx/utils/helper.unit.js` |
| `apps/sie-iidp-xxx/components/Modal.vue` | `unitTest/apps/sie-iidp-xxx/components/Modal.unit.js` |

### 测试文件命名规范

| 规则 | 示例 |
|---|---|
| 后缀固定为 `.unit.js` | `translationManagement.unit.js` |
| 文件名与源码文件名一致 | `helper.js → helper.unit.js` |
| `describe` 块标注模块名 | `describe('translationManagement 模块', ...)` |
| `it/test` 描述使用中文，包含函数名 | `it('filterEmptyString 应正确过滤空字符串', ...)` |

## 三、运行命令大全

### 3.1 全量测试 + 覆盖率

```bash
npm run test:coverage
```

输出位置：`testReport/coverage/index.html`（可在浏览器打开查看）

### 3.2 全量测试 + HTML 报告

```bash
npm run test:report
```

输出位置：`testReport/report/jest-report.html`

### 3.3 ⭐⭐ 按文件运行单元测试

<span style="color:red;font-weight:1000">修改某个具体模块时，只针对该模块的测试文件运行，避免等待全量测试。</span>

**格式：**

```bash
npx jest --config=jest.config.coverage.js <测试文件路径>
```

**示例：**

```bash
# 方式一：直接用 npx
npx jest --config=jest.config.coverage.js unitTest/apps/sie-iidp-i18n-web/views/xxx/xxx.unit.js

# 方式二：使用 npm 脚本（已在 package.json 配置）
npm run test:unit
```

### 3.4 ⭐⭐ 按函数名运行单元测试

<span style="color:red;font-weight:1000">使用 Jest 的 `-t`（`--testNamePattern`）参数，精确定位某个 `it/test` 用例。方法变更时只验证该方法，无需跑全文件。</span>

**格式：**

```bash
npx jest --config=jest.config.coverage.js -t "<函数/用例名称>" <测试文件路径>
```

**示例：**

```bash
# 只跑 filterEmptyString 相关用例
npx jest --config=jest.config.coverage.js -t "filterEmptyString" unitTest/apps/sie-iidp-i18n-web/views/translationManagement/translationManagement.unit.js

# 只跑描述中含"导出"的用例
npx jest --config=jest.config.coverage.js -t "导出" unitTest/apps/sie-iidp-i18n-web/views/translationManagement/translationManagement.unit.js

# 支持正则：匹配所有以 "should" 开头的用例
npx jest --config=jest.config.coverage.js -t "^should" unitTest/apps/sie-iidp-i18n-web/views/translationManagement/translationManagement.unit.js
```

**Mac / Linux 通过环境变量调用（package.json 已配置 test:func）：**

```bash
FUNC="filterEmptyString" \
FILE="unitTest/apps/sie-iidp-i18n-web/views/translationManagement/translationManagement.unit.js" \
npm run test:func
```

> **注意：** `-t` 匹配的是 `it()/test()` 的**描述字符串**，不是函数名本身。  
> 推荐命名规范：`it('函数名 应该...', ...)` ，这样用函数名即可精确过滤。

### 3.5 按扩展 / 模块目录运行

当一个扩展下有多个测试文件时，可按目录批量运行：

```bash
# 运行整个扩展的所有测试
npx jest --config=jest.config.coverage.js unitTest/apps/sie-iidp-i18n-web/

# 运行特定功能目录
npx jest --config=jest.config.coverage.js unitTest/apps/sie-iidp-i18n-web/views/translationManagement/

# 监听模式（TDD 开发时实时反馈）
npx jest --watch --config=jest.config.coverage.js unitTest/apps/sie-iidp-i18n-web/
```

### 3.6 命令速查表

| 场景 | 命令 | 适用时机 |
|---|---|---|
| 全量测试 + 覆盖率 | `npm run test:coverage` | 发版前、PR 合并前 |
| 全量测试 + HTML 报告 | `npm run test:report` | 定期汇报、演示 |
| 单文件测试 | `npm run test:unit` | 开发某个模块时 |
| 单函数测试 | `npx jest ... -t "函数名" 文件` | 修改单个方法后验证 |
| 单扩展测试 | `npx jest ... unitTest/apps/扩展名/` | 扩展级别回归 |
| 监听模式 | `npx jest --watch ... 文件` | TDD 开发时实时反馈 |

## 四、变更流程场景

代码发生变更时，必须联动对应单元测试。以下是典型场景的操作规范。

### 场景 1：新增一个工具函数

| 步骤 | 操作说明 |
|---|---|
| 1. 编写源码 | 在 `apps/` 对应目录新增函数 |
| 2. 创建测试文件 | 在 `unitTest/` 镜像路径新建 `xxx.unit.js` |
| 3. 编写测试用例 | 覆盖正常值、边界值、异常值 |
| 4. 运行单文件测试 | `npx jest --config=... xxx.unit.js` |
| 5. 检查覆盖率 | 确保行覆盖率 ≥ 90% |
| 6. 提交代码 | `git add + git commit`（含测试文件） |

### 场景 2：修改已有方法

| 步骤 | 操作说明 |
|---|---|
| 1. 定位影响范围 | 确认修改了哪个函数/方法 |
| 2. 运行函数级测试 | `npx jest ... -t "方法名" 对应.unit.js` |
| 3. 若测试失败 | 更新测试用例以反映新逻辑 |
| 4. 追加新场景用例 | 为新逻辑补充测试覆盖 |
| 5. 回归整个文件 | `npx jest --config=... 整个.unit.js` |
| 6. 检查覆盖率报告 | 确保覆盖率无下降 |

### 场景 3：重构模块 / 拆分文件

| 步骤 | 操作说明 |
|---|---|
| 1. 重构源码 | 保持函数签名或同步更新 |
| 2. 同步调整测试文件 | 根据新路径/新函数名更新 import 和用例 |
| 3. 运行受影响扩展测试 | `npx jest ... unitTest/apps/扩展名/` |
| 4. 全量覆盖率验证 | `npm run test:coverage` 确认无退化 |
| 5. Code Review 包含测试 | 提交 PR 时测试文件纳入 Review |

### 场景 4：删除功能

| 步骤 | 操作说明 |
|---|---|
| 1. 删除源码 | 移除对应函数/文件 |
| 2. 同步删除测试文件 | 删除 `unitTest/` 下对应 `.unit.js` |
| 3. 全量测试确认 | 运行 `npm run test:coverage` 确认无引用错误 |

### 场景 5：第三方依赖升级

| 步骤 | 操作说明 |
|---|---|
| 1. 升级依赖 | `npm update xxx` |
| 2. 运行全量测试 | `npm run test:coverage` |
| 3. 分析失败用例 | 根据报告定位受影响函数 |
| 4. 修复 Mock 或断言 | 依赖 API 变更时更新 mock 实现 |
| 5. 输出覆盖率分析报告 | 若覆盖率下降，喂给 AI 出分析报告（见第七章） |

### 变更流程决策树

```
代码变更？
  ├─ 新增函数    →  新建 .unit.js，写测试，覆盖率 ≥ 90%
  ├─ 修改函数    →  -t "函数名" 跑单函数测试，更新用例
  ├─ 重构模块    →  同步更新 unitTest/，跑扩展级测试
  ├─ 删除功能    →  删除对应 .unit.js，全量测试确认
  └─ 依赖升级    →  全量测试 → 失败分析 → AI 分析报告
```

## 五、AI 辅助编写单元测试

AI 工具可大幅降低编写测试用例的成本。根据是否付费，推荐以下工具和对应操作方式。

### 5.1 方案对比总览

| 对比项 | 免费方案（Trae 国际版/国内版） | 付费方案（Cursor / Claude Pro） |
|---|---|---|
| 使用方式 | IDE 内直接调用，无需粘贴代码 | IDE 内直接调用，无需粘贴代码 |
| 上下文长度 | 有限，大文件建议按模块拆分 | 超长上下文（200K+），整个文件一次性分析 |
| 多轮对话 | 支持，但上下文窗口较小 | 记忆完整，可连续迭代 |
| 生成质量 | 良好，适合单文件场景 | 更精准，贴合项目整体约定 |
| 适合场景 | 日常开发、单个模块补测 | 批量生成、覆盖率攻坚、复杂逻辑 |
| 成本 | **免费** | 约 $20/月 或企业授权 |

### 5.2 ⭐⭐ 通用提示词（核心）

无论使用哪种工具，以下提示词均可直接使用。将其发送给 AI，AI 会自动扫描文件并生成测试。

```
扫描 apps 文件夹下的 JS 文件，为每个文件在 unitTest/ 目录下的镜像路径生成对应的单元测试文件（后缀 .unit.js），要求如下：

1. 扫描 apps 文件夹的所有 .js / .ts 文件，在 unitTest/ 目录按相同路径结构生成 .unit.js 测试文件
2. 行覆盖率必须达到 90%；如果当前生成的测试未达到 90%，自动补充用例直到满足要求
3. 每个测试用例需要有中文注释，说明该用例的测试目的
4. 如果源码中使用了 window、document、localStorage、sessionStorage、ResizeObserver、IntersectionObserver 等浏览器 API，在测试文件顶部自动添加对应的 jest.fn() mock
5. 全局 mock 变量（Tech、tech、window.ELEMENT 等）已在 unitTest/jest.setup.js 中定义，测试文件无需重复 mock，直接使用即可
6. 测试文件结构要求：
   - describe 块使用模块名作为描述
   - 每个 it() 描述用中文，格式：it('函数名 应该...', ...)
   - 包含正常值、边界值、异常值三类用例
7. 输出完整可直接运行的测试文件内容
```

### 5.3 免费方案：Trae 国际版/国内版

**Trae**是免费的 AI IDE，支持在编辑器内直接与 AI 对话，无需手动复制代码，适合日常开发中对单个模块补充单元测试。

**操作步骤：**

1. 在 Trae 中打开项目，定位到需要补测的源码文件
2. 打开右侧 AI 对话面板（或使用快捷键唤起）
3. 将上方「通用提示词」粘贴进去，按需补充具体文件路径或函数名
4. AI 生成测试文件内容后，在 `unitTest/` 对应路径创建文件并粘贴
5. 运行测试查看覆盖率：

```bash
npx jest --config=jest.config.coverage.js unitTest/apps/<对应路径>/<文件名>.unit.js
```

6. 若覆盖率未达到 90%，将终端输出的覆盖率摘要粘贴给 AI，继续追问：

```
当前行覆盖率为 XX%，未覆盖的行如下：[粘贴未覆盖信息]
请补充测试用例，使行覆盖率达到 90% 以上。
```

7. 重复步骤 5～6 直到覆盖率 ≥ 90%

> 按单个文件运行单元测试（中等复杂度的文件），用免费ai执行过程大概2分钟；按单个方法单元测试（中等复杂度的方法），用免费ai执行过程大概1分钟。

### 5.5 覆盖率迭代策略

| 覆盖率区间 | 操作建议 |
|---|---|
| < 60% | 检查测试文件是否正确 `require` 了源码；重新让 AI 生成 |
| 60% ~ 80% | 将终端覆盖率输出粘贴给 AI，要求补充边界值和异常分支 |
| 80% ~ 90% | 将未覆盖行号告知 AI，或上传覆盖率 HTML 截图，请 AI 定向补充 |
| ≥ 90% | ✅ 通过！提交代码 |

## 六、按文件统计覆盖率 Excel 输出

### Step 1：开启 JSON 格式输出

在 `jest.config.coverage.js` 中追加 `json-summary`：

```js
coverageReporters: ['html', 'text-summary', 'json-summary'],
```

运行后会在 `testReport/coverage/` 下生成 `coverage-summary.json`。

### Step 2：安装 xlsx 依赖

```bash
npm install xlsx --save-dev
```

### Step 3：创建转换脚本

新建 `scripts/coverage2excel.js`：

```js
const fs   = require('fs');
const XLSX = require('xlsx');

const summary = JSON.parse(
  fs.readFileSync('./testReport/coverage/coverage-summary.json', 'utf8')
);

const rows = [['文件路径', '语句覆盖%', '分支覆盖%', '函数覆盖%', '行覆盖%', '是否达标']];

Object.entries(summary).forEach(([file, data]) => {
  if (file === 'total') return;
  const linePct = data.lines.pct;
  rows.push([
    file.replace(process.cwd() + '/', ''),
    data.statements.pct,
    data.branches.pct,
    data.functions.pct,
    linePct,
    linePct >= 90 ? '✅ 达标' : '❌ 未达标'
  ]);
});

const ws = XLSX.utils.aoa_to_sheet(rows);
ws['!cols'] = [{ wch: 60 }, { wch: 12 }, { wch: 12 }, { wch: 12 }, { wch: 12 }, { wch: 10 }];
const wb = XLSX.utils.book_new();
XLSX.utils.book_append_sheet(wb, ws, '覆盖率统计');
XLSX.writeFile(wb, './testReport/coverage-report.xlsx');
console.log('✅ Excel 已生成：testReport/coverage-report.xlsx');
```

### Step 4：运行生成

```bash

npm run test:coverage && node scripts/coverage2excel.js
```

## 九、常见问题 FAQ

### Q1：运行测试报 "Cannot find module" 错误

**原因：** 测试文件的 require/import 路径不正确。

**解决：** 检查相对路径，或使用 `moduleNameMapper` 配置别名：

```js
// jest.config.coverage.js 追加
moduleNameMapper: {
  '^@/(.*)$': '<rootDir>/apps/xxx/$1'
}
```

### Q2：覆盖率为 0%，所有行都未覆盖

**原因：** `collectCoverageFrom` 路径不匹配实际文件路径，或测试文件没有正确 import 源码。

**解决：** 检查 glob 规则，并确认 `.unit.js` 文件中有 `require` 对应源码文件。

### Q3：window is not defined

**原因：** `testEnvironment` 未设置为 `jsdom`。

```js
// jest.config.coverage.js
testEnvironment: 'jsdom',  // 必须设置
```

### Q4：`-t` 按函数名过滤无效，仍然跑了全部用例

**原因：** `-t` 参数匹配的是 `it()/test()` 的描述字符串，不是函数名本身。

**解决：** 确认 `it()` 描述中包含目标关键词：

```js
// ✅ 推荐：描述中包含函数名
it('filterEmptyString 应过滤空字符串', () => { ... })

// 然后用函数名精确过滤：
npx jest --config=... -t 'filterEmptyString' xxx.unit.js
```

### Q5：jest-stare 报告没有自动打开

**原因：** CI 环境无法打开浏览器。

**解决：** CI 中将 `openReportInBrowser` 设置为 `false`，本地开发时保持 `true`。

### Q6：如何在 CI/CD 中集成？

```yaml
# .github/workflows/test.yml
jobs:
  unit-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: npm ci
      - run: npm run test:coverage
      - uses: actions/upload-artifact@v3
        with:
          name: coverage-report
          path: testReport/coverage/
```
